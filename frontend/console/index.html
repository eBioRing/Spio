repo-hosted frontend

spio control console

The control console is the human-facing page. Registry hosting and cloud platform behavior stay behind backend contracts.

Surface split Console backend surface Registry server contract Cloud control-plane contract

Frontend plane

Control console

  • Workspace inventory and lifecycle
  • Registry and publish visibility
  • Toolchain, dependency, and deployment controls
  • Execution lane and worker-pool summaries

Backend plane A

Repository hosting

  • Immutable blobs and index entries
  • Read root, write root, and promotion flow
  • Package publish transport
  • Static or HTTP(S) registry serving

Backend plane B

Cloud platform

  • Hosted workspace lifecycle
  • Project graph and toolchain routes
  • Dependency, execution, and deployment routes
  • Future queue and worker-pool orchestration

Backend engine

Native core

  • Machine-readable CLI contracts
  • Domain serializers and workflow orchestration
  • Registry client and server modules
  • Cloud policy rendering

backend contract families

Console-facing routes

POST /workspaces/open GET /workspaces/{id}/project-graph POST /workspaces/{id}/tool/* POST /workspaces/{id}/dependencies/* POST /workspaces/{id}/execution/* POST /workspaces/{id}/deployment/* GET /spio-registry.json GET /index/<namespace>/<name>/<version>.json

Non-goals

This frontend does not parse manifests, resolve dependencies, publish packages, or schedule workers. Those stay in backend services and native domain code.

Source of truth

Contract truth lives in docs/governance/, docs/registry/, and src/. This page is a concrete frontend scaffold, not a second backend spec.
